Vous avez déjà appris comment utiliser MEI ProfileDrafter, et vous avez utilisé le Service de personnalisation MEI pour compiler votre propre personnalisation. Vous avez téléchargé un fichier .odd avec cette personnalisation. Nous supposons que votre personnalisation exclut certains modules, éléments et classes d'attributs. Examinons maintenant comment cela se manifeste dans le fichier de personnalisation ODD. Ouvrez le avec l'éditeur XML de votre choix.

Vous trouverez un fichier TEI, avec un avertissement en commentaire, dans les premières lignes du fichier, et un élément <teiHeader> qui peut servir de point de départ pour fournir une documentation appropriée pour votre personnalisation. Vous pouvez ajuster le <respStmt> (déclaration de responsabilité) qui se trouve généralement vers la ligne 27 pour revendiquer la responsabilité de cette personnalisation. Le contenu qui nous intéresse commence à la ligne 54 (le numéro exact peut varier). Là, nous avons l'élément <schemaSpec> qui commence notre personnalisation. Mettons de côté les attributs spécifiés ici, et ignorons également l'élément <constraintSpec> pour le moment.

Ce que nous voyons ensuite est une liste d'éléments <moduleRef> (référence de module), chacun se référant à un module MEI avec son attribut @key. Seuls les attributs que vous n'avez pas exclus du schéma sont répertoriés ici. En conséquence, il est très simple d'exclure manuellement un module MEI d'une personnalisation : vous devez simplement supprimer la référence à celui-ci dans votre fichier de personnalisation.

Dans un ou plusieurs <moduleRef>, vous devriez voir un attribut supplémentaire @except. Cet attribut contient une liste d'éléments du module spécifié qui doivent être exclus de la personnalisation. En d'autres termes, il permet de sélectionner précisément les éléments à retirer, tout en conservant tous les autres éléments du module dans votre personnalisation. ODD propose également l'approche inverse en permettant d'autoriser explicitement des éléments d'un module grâce à l'attribut @include. Il n'est toutefois pas permis de combiner les deux approches. Conformément aux directives TEI, les processeurs ODD signaleront cela comme une erreur. Lors de l'importation de personnalisations existantes, MEI ProfileDrafter reconnaît ces deux méthodes, mais lors de l'exportation des données, il privilégiera l'attribut @except plutôt que @include

Enfin, à la fin de votre fichier ODD, vous trouverez une liste d'éléments <classSpec> (spécification de classe) avec @type="atts". Ces éléments utilisent également l'attribut @ident pour identifier des classes d'attributs individuelles. Notez que cet élément ne contient aucune référence au module dans lequel l'attribut est défini. À la place – et c'est plus important – il utilise l'attribut @mode pour spécifier ce qui doit être fait aux sources MEI originales, qui sont censées être modifiées par votre personnalisation. Ici, la seule valeur utilisée sera @mode="delete" – les classes d'attributs correspondantes seront supprimées de votre personnalisation, les rendant indisponibles dans le schéma résultant. Encore une fois, il est très simple de faire de telles exclusions manuellement.

Comme vous pouvez le constater, le MEI ProfileDrafter fait en réalité très peu de choses dans ODD – c'est principalement une interface utilisateur pratique qui aide à illustrer les conséquences de la suppression de divers composants du schéma. Par exemple, il montre très clairement quels attributs sur l'élément <note> seront indisponibles lorsque le module MEI.gestural est désactivé. Techniquement, cependant, il ne fait que des choses très simples que l'on pourrait mettre en œuvre manuellement tout aussi bien.
